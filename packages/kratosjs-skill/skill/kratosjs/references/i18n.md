# Internationalization

i18n is configured **once on the backend** — it is the single source of truth. The server injects the config into the page (`window.__VALAJS_I18N__`) and the React client auto-configures itself. There is no separate client i18n setup.

```ts
adminPanel.i18n({ locales: ['en', 'sq'], defaultLocale: 'en', fallbackLocale: 'en' }).registerTranslations('app', {
	en: { 'users.label': 'Users' },
	sq: { 'users.label': 'Përdoruesit' },
});
```

- `.i18n({ locales, defaultLocale, fallbackLocale })` declares the available languages.
- `.registerTranslations(namespace, catalogsByLocale)` adds message catalogs. Call it per namespace (e.g. `'app'`, or a plugin name).
- The client reads the injected config and switches languages without extra wiring.

## Server-side translation

Server code can translate with the exported helpers: `t(...)`, `withLocale(...)`, and locale resolution utilities (`resolveServerLocale`, `resolveRequestLocale`). These are server-only (they use `AsyncLocalStorage`).

## Notes

- Because config is backend-owned, don't try to configure i18n in `main.tsx` — register locales and translations on the panel instead.
- Core resource/UI strings ship with the framework; add your app- and plugin-specific strings via `registerTranslations`.
